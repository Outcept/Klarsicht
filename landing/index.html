The problem

You already know this drill

Every on-call engineer has lived this. The page comes in, and the clock starts ticking.

03:12 PagerDuty fires. Pod CrashLoopBackOff in production.
03:14 SSH into bastion, switch kubecontext. Which namespace again?
03:22 kubectl logs, kubectl describe. Scroll through walls of text looking for clues.
03:41 Open Grafana. Compare metrics. Check if the deploy happened before or after the spike.
04:08 Ask a colleague in Slack. They're asleep.
05:15 Finally find the root cause. 2 hours of context-switching, half-asleep debugging.

How it works

From alert to root cause in three steps

Point your Grafana webhook at Klarsicht. The AI agent handles the rest.

01

Alert fires

Grafana sends a webhook to Klarsicht when any alert triggers. No new alert rules needed — your existing setup works as-is.

02

Agent investigates

The AI agent inspects pods, reads logs, pulls events, and queries Prometheus metrics via the Kubernetes API. Like a senior SRE, but instant.

03

RCA delivered

A structured root cause analysis with a timeline, impact assessment, fix steps, and a postmortem draft lands in your dashboard within seconds.

─── Root Cause Analysis ───────────────────────────────

root_cause:
  The postgres-0 pod was OOMKilled due to a memory spike
  caused by an overnight batch job (cron-etl-daily) that
  increased query volume by 340%. The api-gateway pod
  entered CrashLoopBackOff because it cannot start without
  a Postgres connection.

chain_of_events:
  01:15  cron-etl-daily job started, issuing heavy queries
  03:09  postgres-0 memory hit 509Mi/512Mi limit, OOMKilled
  03:09  postgres service endpoints dropped to 0
  03:10  api-gateway failed health check, restarted
  03:12  api-gateway entered CrashLoopBackOff after 3 failures

impact:
  API gateway unreachable. All downstream services returning
  502. Estimated user impact: 100% of API traffic.

fix_steps:
  1. Increase postgres-0 memory limit to 1Gi
     kubectl patch statefulset postgres -n production \
       -p '{"spec":{"template":{"spec":{"containers":[{"name":"postgres","resources":{"limits":{"memory":"1Gi"}}}]}}}}'
  2. Restart api-gateway deployment
     kubectl rollout restart deployment/api-gateway -n production
  3. Schedule cron-etl-daily during off-peak hours
  4. Add PodDisruptionBudget for postgres StatefulSet

Features

Everything you need, nothing you don't

Kubernetes-native

Reads pods, logs, events, and deployments directly via the K8s API. Understands your cluster topology, not just individual resources.

Prometheus & Mimir compatible

Queries PromQL to detect anomalies in CPU, memory, network, and custom metrics. Works with Prometheus, Mimir, or any compatible backend.

Structured postmortems

Generates a timeline, impact assessment, and action items. Copy as Markdown, paste into your incident channel or wiki. Done.

Self-hosted

Runs inside your cluster. Logs and metrics never leave your infrastructure. Meets data residency requirements out of the box.

Helm install

One command, five minutes. Deploys into your existing monitoring stack alongside Grafana and Prometheus. No migration needed.

Works with existing alerts

Point a Grafana webhook contact point at Klarsicht. Your current alert rules, thresholds, and routing stay exactly the same.

Get started

Deploy in five minutes

Three commands. Three environment variables. Your first root cause analysis in under a minute.

1. Add the Helm repo and install

# Add the Klarsicht Helm repository helm repo add klarsicht https://gitlab.com/api/v4/projects/outcept%2Fklarsicht/packages/helm/stable helm repo update # Install into your cluster helm install klarsicht klarsicht/klarsicht \ --namespace klarsicht \ --create-namespace \ --set config.grafanaWebhookSecret=<your-webhook-secret> \ --set config.llmApiKey=<your-api-key> \ --set config.prometheusUrl=http://prometheus.monitoring:9090

2. Configure Grafana contact point

# In Grafana: Alerting → Contact points → New contact point # Type: Webhook # URL: http://klarsicht.klarsicht.svc.cluster.local:8080/api/v1/webhook/grafana

3. Trigger an alert or send a test

# Send a test alert from Grafana, or use the CLI: kubectl port-forward svc/klarsicht -n klarsicht 8080:8080 curl -X POST http://localhost:8080/api/v1/test-alert
